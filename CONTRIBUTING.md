# Contributing to Dev → Product Manager

This is a practical, honest knowledge base written for developers making the transition to Product Management. Contributions are welcome — especially if you've lived it.

## What makes a good contribution

- **Developer-centric voice.** Assume the reader codes for a living and is new to product. Use engineering analogies. Don't over-explain what a for-loop is; do explain what a P&L is.
- **Honest takes.** If a framework is overrated, say so. If something is hard, say so. No marketing language.
- **Concrete over abstract.** Examples, templates, and checklists beat general principles.
- **Short over long.** If something can be said in one sentence, don't use five.

## What to contribute

- Corrections to factual errors
- Missing topics (open an issue first so we can agree on scope)
- War stories from your own dev→PM transition — real examples make the best teaching material
- Templates you've actually used
- Resources (books, courses, communities) you've personally found valuable

## What not to contribute

- Certification promos or affiliate content
- Content that's already well-covered in the existing pages
- Framework hagiography — every framework has failure modes; mention them

## How to contribute

1. Fork the repo
2. Create a branch: `git checkout -b add/your-topic`
3. Write or edit content following the style guidelines below
4. Open a pull request with a short description of what you changed and why

## Style guidelines

- **Headers:** `## Title` for major sections, `### Subsection` for sub-topics
- **Tables:** use for comparisons and structured lists
- **Code blocks:** use for templates, command examples, and structured formats (even if not actual code)
- **Length:** content pages 2,000–5,000 words; templates 300–800 words
- **Tone:** second person ("you"), present tense, active voice

## Questions?

Open an issue. We'll get back to you.
